# エフ・キャンカップ 謎解きマッチングサイト構成案

## 目的

高校生・大学生がサッカーイベント会場で謎解きゲームを楽しみながら、協賛企業の事業内容・仕事・魅力を知り、企業との接点を作るWebサイトを作る。

謎解きは「企業をヒーローとして呼び覚ます」ストーリーにし、各企業の情報を調べることで答えにたどり着く形式にする。

## 想定ユーザー

- 高校生・大学生の参加者
- 小学生参加者の保護者
- 協賛企業の担当者
- イベント運営スタッフ

## サイト全体の構成

### 1. トップページ

役割: イベントの入口。

掲載内容:

- イベント名
- キャッチコピー
- 開催日時・場所
- 参加対象
- 謎解きゲーム開始ボタン
- 企業マッチングの説明
- 注意事項

導線:

- 「謎解きを始める」
- 「参加登録」
- 「協賛企業を見る」

## 2. 参加登録ページ

役割: 参加者を識別し、進行状況やマッチング結果を保存する。

入力項目:

- 名前
- 学校名
- 学年
- メールアドレス
- 興味のある分野
- サッカー経験またはイベント参加理由

最初のMVPではGoogleフォームでも代用可能。

将来的にはSupabaseやFirebaseに保存する。

## 3. ストーリーページ

役割: 謎解きの世界観を伝える。

内容:

- 未来の函館に巨大怪獣が出現
- 街を守っていた企業ヒーローたちは忘れられ、力を失っている
- 参加者はスマホと知恵を使って、企業ヒーローの本当の姿を突き止める
- 全てのヒーローを集めると怪獣に立ち向かえる

ボタン:

- 「ミッションを開始する」

## 4. ミッション一覧ページ

役割: 企業ごとの謎解き入口。

表示内容:

- 企業名
- ヒーロー名
- ミッション状態
  - 未挑戦
  - 挑戦中
  - クリア
- 難易度
- 企業ジャンル

例:

- 翔大鋼業: 鉄の骨組みを操るヒーロー
- 函館中央薬局: 街の健康を守るヒーロー
- 森川組: 街を創るヒーロー
- 函館どつく: 船と橋を支えるヒーロー
- 大明工業: 技術で暮らしを支えるヒーロー
- 東興アイテック: 産業を支えるヒーロー

## 5. 企業別ミッションページ

役割: 謎解きの中心ページ。

構成:

- ストーリー導入
- 企業調査ミッション
- 謎解き問題
- 回答入力欄
- ヒントボタン
- 正解後の企業紹介
- 「この企業に興味がある」ボタン

例: 翔大鋼業ミッション

- 企業HPを調べて「鉄筋工事」を見つける
- 鉄筋工事の説明を読む
- 文字パズルを解いて「技術力」を導く
- 正解後に、翔大鋼業の仕事・魅力・求める人物像を表示

## 6. 企業紹介ページ

役割: 謎解きで興味を持った企業を深く知る。

掲載内容:

- 企業名
- 業種
- 事業内容
- 会社の強み
- 高校生・大学生に伝えたいこと
- 働く人のインタビュー
- 写真・動画
- 公式サイトリンク
- ブース位置
- 「話を聞きたい」ボタン

## 7. マッチング入力ページ

役割: 学生の興味を企業につなげる。

入力項目:

- 興味のある企業
- 話を聞きたい企業
- 興味のある仕事
- 希望する接点
  - 当日ブースで話したい
  - 後日説明を聞きたい
  - インターン情報がほしい
  - 会社見学に興味がある
- 自由記述

## 8. クリア画面

役割: 体験の達成感を作る。

表示内容:

- 集めたヒーロー一覧
- 最終ストーリー
- 怪獣撃退演出
- クリア証
- 興味を持った企業一覧
- 次の行動ボタン
  - 企業ブースに行く
  - アンケートに答える
  - 結果を保存する

## 9. 管理者ページ

役割: 運営側が参加者やマッチング状況を確認する。

必要機能:

- 参加者一覧
- 企業別の興味あり人数
- 参加者ごとのクリア状況
- 企業ごとのマッチング候補
- CSV出力
- 回答ログ確認

MVPではGoogleスプレッドシートで代用可能。

## 必要な機能一覧

### MVPで作る機能

- トップページ
- ストーリーページ
- 企業別ミッションページ
- 回答入力と正誤判定
- ヒント表示
- 企業紹介
- 興味ありボタン
- 最後のアンケートフォーム

### 余裕があれば作る機能

- 参加者登録
- 進行状況の保存
- クリア証発行
- 企業別ランキング
- 管理画面
- 企業担当者用の閲覧ページ

## サーバーの必要性

### サーバーなしで作れる範囲

以下だけなら静的サイトで作れる。

- 問題表示
- 回答入力
- 正誤判定
- ヒント表示
- 企業紹介
- Googleフォームへの送信

使用候補:

- HTML / CSS / JavaScript
- GitHub Pages
- Netlify
- Vercel

### サーバーまたはバックエンドが必要な範囲

以下をやるなら必要。

- 参加者ごとの進行状況保存
- 学生情報の保存
- 企業への興味データ保存
- 管理画面
- マッチング結果の集計
- 個人情報の安全な管理
- 回答を完全に隠す

使用候補:

- Supabase
- Firebase
- Vercel + データベース

## おすすめ構成

### 第1段階: デモ版

目的: イベントのイメージを見せる。

構成:

- 静的Webサイト
- JavaScriptで回答判定
- 企業紹介ページ
- Googleフォームで興味登録

メリット:

- 早く作れる
- サーバー不要
- 費用が少ない
- 企画説明に使いやすい

### 第2段階: イベント運用版

目的: 実際の参加者データを扱う。

構成:

- フロントエンド: React / Next.js
- データベース: Supabase
- 認証: メールまたは簡易コード
- 管理画面: 運営用ダッシュボード

できること:

- 参加者登録
- 進行状況保存
- 企業への興味登録
- マッチング集計
- CSV出力

## デザイン方針

- サッカーイベントらしく、明るく動きのある印象にする
- 企業紹介は信頼感を重視する
- 謎解き部分はゲーム感を出す
- スマホ利用を前提にする
- 会場で迷わないようにボタンを大きくする
- 小学生イベントの場でも違和感がない、健全で前向きな雰囲気にする

## 実装メモ

最初はサーバーなしで作り、後からSupabaseを追加する形がよい。

最初のデモ版では、回答はJavaScript内に持たせる。ただし本番では答えが見られる可能性があるため、重要な判定はバックエンド側に移す。

個人情報を扱う場合は、必ずプライバシーポリシーと利用目的の説明を用意する。

## 結論

一度Webサイトとして作るなら、まずはサーバーなしのデモ版で十分。

ただし、学生と企業のマッチングを本格運用する段階では、SupabaseやFirebaseなどのバックエンドを使う構成にする。
