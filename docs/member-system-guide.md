# メンバー共有用: システム構成と今後の進め方

この資料は、F-CAN CUP / Hero Match Quest をメンバーに引き継ぐためのシステム構成メモです。

## 1. 結論

現在のサイトは、静的サイトとしてかなりシンプルにまとまっています。

```txt
HTML + CSS + JavaScript
GitHub Pages で公開
進行状況は localStorage に保存
```

今の目的が「イベント会場で謎解き体験を見せる」「企業ヒーローの世界観を伝える」「QRコードで参加者に遊んでもらう」であれば、この構成のままで問題ありません。

ただし、以下をやる場合は現在の構成だけでは足りません。

- 個人情報をサーバーに保存する
- 参加者ごとの進行状況を複数端末で共有する
- 管理画面を作る
- 企業別の興味データを集計する
- CSV出力する
- 回答を参加者に絶対見えないようにする

この段階に進む場合は、Firebase などのバックエンドを追加します。

## 2. 現在の全体構成

```txt
参加者のスマホ/PC
  ↓
GitHub Pages
  ↓
index.html
styles.css
app-story.js
assets/hero-quest-map.jpg
  ↓

```

- トップページ表示
- ストーリー表示
- 受付フォーム
- ミッション一覧表示
- 6社の謎解き
- ヒント表示
- 正解判定
- ヒーロー解放
- 企業紹介
- 気になる企業の保存
- 最終決戦演出
- 同じ端末内での進行状況保存

### 現在できないこと

- 他の端末への進行状況引き継ぎ
- 運営側での参加者一覧確認
- 個人情報の安全なサーバー保存
- 管理者ログイン
- 回答の完全な秘匿
- 本格的なマッチング集計

## 5. 今後の進め方

### 第1段階: 現在の静的サイトを完成させる

目的:

- 会場で参加者が迷わず遊べる状態にする
- 企業ヒーローの世界観を強くする
- スマホ表示の崩れをなくす

やること:

- 6社の企業情報を実データに近づける
- 謎解き文を調整する
- ブース番号を確定する
- トップ画像を本番用にする
- スマホ表示を確認する
- GitHub Pages で公開確認する

この段階では、現在の構成のままでよいです。

### 第2段階: イベント運用に必要な受付を決める

まず決めること:

- 名前を集めるか
- 学校名を集めるか
- メールアドレスを集めるか
- 企業への興味データを保存するか
- 誰がデータを見るか
- いつデータを削除するか

個人情報を集めないなら:

```txt
今の localStorage 受付のままでOK
```

個人情報を集めるなら:

```txt
Googleフォーム
または
Firebase / Supabase
```

を使います。

### 第3段階: 管理画面が必要か判断する

管理画面が不要な場合:

- GitHub Pages + Googleフォームで十分
- 運営はGoogleスプレッドシートで確認

管理画面が必要な場合:

- Firebase Authentication
- Firestore
- Security Rules
- 管理者専用画面
- CSV出力

を追加します。

### 第4段階: 本格運用版に拡張する

複数イベントで使い回す、企業担当者も見る、参加者データを保存する場合は、現在の静的サイトから以下の構成に拡張します。

```txt
GitHub
  -> コード管理

Firebase Hosting または GitHub Pages
  -> サイト公開

Firebase Authentication
  -> 管理者ログイン

Cloud Firestore
  -> 参加者情報、進行状況、興味データ保存

Cloud Functions
  -> CSV出力、集計、メール通知
```

## 6. 今すぐのおすすめ方針

今回のサイトは、まず以下の方針で進めるのがよいです。

```txt
1. 今の静的サイトを完成させる
2. 企業情報と謎の内容を確定する
3. GitHub Pages で公開する
4. 会場テストをする
5. 個人情報を集めるかどうかを判断する
6. 必要なら Googleフォーム or Firebase を追加する
```

理由:

- 今の構成は軽い
- スマホで開きやすい
- サーバー管理が不要
- イベント前の修正が早い
- まず体験の完成度に集中できる

## 7. 注意すべきこと

### 回答は完全には隠せない

現在の答えは `app-story.js` に入っています。

普通の参加者は見ませんが、詳しい人がコードを見れば答えは分かります。

イベント体験としては問題ありません。
ただし、不正防止が重要なコンテスト形式にするなら、回答判定をサーバー側に移す必要があります。

### localStorage は同じ端末だけ

現在の進行状況は、同じスマホ・同じブラウザ内だけに保存されます。

ブラウザを変える、端末を変える、データを消すと進行状況は引き継げません。

### 個人情報は慎重に扱う

現在の受付は、サーバー送信しない簡易受付です。

メールアドレス、学校名、本名などを運営が保存する場合は、以下を決めます。

- 利用目的
- 保存先
- 閲覧できる人
- 削除期限
- 問い合わせ先

## 8. メンバーにお願いしたい運用ルール

### 編集前

- 何を直すか決める
- 触るファイルを確認する
- 大きな変更を一気に入れない

### 編集後

- `index.html` をブラウザで開いて確認する
- スマホ幅でも確認する
- 受付、ミッション、最終決戦の導線を確認する
- Gitの差分を確認する

### 公開前

- トップページが表示されるか
- 6社のカードが表示されるか
- 答えを入力して解放できるか
- 最終決戦がロック/解放されるか
- 不要な画像や古いファイルが混ざっていないか

## 10. まとめ

現在のシステムは、イベント用の体験サイトとしては十分に軽く、扱いやすい構成です。

当面はこのまま静的サイトとして完成度を上げます。

次に個人情報、管理画面、企業別集計が必要になった段階で、Firebase などのバックエンドを追加します。

今すぐバックエンド化するより、まずは以下に集中するのがよいです。

- 企業情報の正確さ
- 謎解きの分かりやすさ
- スマホでの見やすさ
- 会場で迷わない導線
- ブースに行きたくなる企業紹介
